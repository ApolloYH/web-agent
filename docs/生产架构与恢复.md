# Apollo Web Agent 生产架构与恢复

本文描述当前代码真实支持的生产形态。目标是把单节点部署做稳、做清楚；没有实现的多机能力不写成“高可用”。

## 1. 支持的拓扑

```text
Browser
  ├─ HTTPS ─> Nginx / FRP ─> Node :9130
  │                         ├─ React 静态资源
  │                         ├─ 登录、对话、文件和 SSE API
  │                         ├─ SQLite
  │                         └─ 用户工作区
  └─ HTTPS ─> Office CDN ─> Office Host :5174
                            └─ OnlyOffice Browser 运行时

Node ─> OpenAI-compatible LLM / LangCore / LangHub
```

- 主站进程是唯一写入者，SQLite 使用 WAL；同步 SQL 操作保持短小，密码散列和外部请求均不占用数据库事务。
- Office Host 只提供静态编辑器运行时。DOCX 内容留在浏览器或用户工作区，不交给 OnlyOffice DocumentServer。
- 当前正式支持单应用实例。多实例需要先替换 SQLite、进程内任务锁和本地文件系统，不能只增加 Nginx upstream。

## 2. 隔离边界

- 每个账号拥有独立的 `workspace`、Apollo Engine、session、个人记忆和成果目录。
- 统一入口每个对话拥有独立 session；相同 LangHub 项目因共享远端工作区而串行，不同项目可并行。
- Web 永久拒绝 `shell_exec`、工作区越界、敏感文件读取、Skill 安装及 MCP 调用。`WEB_ALLOW_UNRESTRICTED=true` 只允许界面进入免审批模式，不解除这些 Web 边界。
- 非管理员的 Skill 目录固定为 `./assistant-skills`，MCP 配置会被清空；完整运行时 JSON 只有管理员可读写。
- 浏览器本地目录句柄保存在浏览器 IndexedDB。手动打开和编辑不上传原文件；只有用户发起云端问答/修改时，目录清单或当前文档正文才通过显式编辑器桥传给 Agent。

## 3. 写入、一致性与任务状态

- 网站文件使用 ETag/`If-Match`，发现外部新版本时返回冲突，不静默覆盖。
- Markdown/JSON 保存按队列串行；关闭文档前取消防抖并等待最后一次写入。Word 关闭前调用编辑器原生 DOCX 导出并等待保存回调。
- 每个 Agent 请求写入 `agent_runs`：`running → succeeded/failed/cancelled`。启动时遗留的 `running` 会标记为 `interrupted`。
- 服务重启后不会自动重放 Agent 任务。外部工具可能产生不可逆副作用，自动重放会重复提交；用户应根据运行记录明确重试。
- 默认全局 8 个 Agent 任务、每账号 3 个；文件传输全局 2 路、每账号 1 路。超限返回 `429` 和 `Retry-After`。

## 4. 存储边界

- 单次上传最多 8 个文件、请求最多 64MB、单文件最多 20MB；单个 LangHub/导入成果最多 25MB。
- 默认保留至少 512MB 磁盘空间；低于阈值时 readiness 返回 503 并拒绝新文件写入。
- 默认每个用户工作区最多 2GB；上传、成果导入和 LangHub 下载落盘前都会校验。
- `uploads/` 默认保留 7 天，成果文件不按该策略清理。上述阈值分别由 `WEB_MIN_FREE_DISK_BYTES`、`WEB_USER_STORAGE_QUOTA_BYTES`、`WEB_UPLOAD_RETENTION_DAYS` 调整。

## 5. 健康检查与运维判断

- `GET /livez`：只证明 Node 事件循环仍可响应，不访问数据库或 Office Host。
- `GET /healthz`：检查服务关闭状态、SQLite 查询、磁盘余量和 Office Host；适合部署验收和流量入口 readiness。
- 响应包含 `X-Request-ID`；Agent SSE 另外包含 `X-Apollo-Run-ID`，可与日志和 `agent_runs` 对齐。
- `/healthz` 失败时先看 `databaseReady`、`diskFreeBytes`、`officeReady`，不要用无条件重启掩盖磁盘耗尽。

## 6. 备份

备份脚本使用 SQLite 在线 backup API，避免直接复制 WAL 中的数据库。它保存 `.apollo` 和存在时的 `.env`，跳过可重新生成的 `onlyoffice-runtime`，快照先写临时目录再原子改名。

```bash
node scripts/backup-production.mjs \
  --source /opt/wyd-web-agent/shared \
  --destination /opt/wyd-web-agent/backups \
  --retention-days 14
```

生产 timer：

```bash
sudo cp ops/web-agent-backup.{service,timer} /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now web-agent-backup.timer
sudo systemctl start web-agent-backup.service
sudo systemctl status web-agent-backup.service web-agent-backup.timer
```

备份包含密钥和用户文件，目录必须只允许 `wyd-agent` 与管理员读取。生产部署后至少执行一次备份并打开快照中的 `manifest.json`；只看到 timer 为 active 不等于备份可恢复。

## 7. 恢复

恢复必须在应用停止时进行：

```bash
sudo systemctl stop web-agent.service
node scripts/restore-production.mjs \
  --snapshot /opt/wyd-web-agent/backups/<timestamp> \
  --target /opt/wyd-web-agent/shared \
  --confirm
sudo systemctl start web-agent.service
curl -fsS http://127.0.0.1:9130/healthz
```

默认不恢复 `.env`；确需恢复密钥时追加 `--restore-env`。旧数据保留为 `.apollo.pre-restore-<timestamp>`，验收完成后再人工删除。恢复后应验证登录、历史对话、一个文件下载和一次 Word 保存。

## 8. 明确边界

- 扫描型 PDF 没有文本层时，当前不做 OCR。
- 图片问答当前只能读取尺寸、类型和大小，不做视觉内容识别。
- Agent 对 Word 的自动修改是正文级查找替换/追加，不是结构化表格或样式编辑；人工仍可用内嵌编辑器完整编辑。
- 少数包含复杂嵌入对象的 DOCX 会触发浏览器 X2T 兼容性错误，此时安全降级为只读预览且不改原文件。
- 真正的跨节点高可用需要外部关系数据库、对象存储、分布式任务租约和共享会话状态，是下一阶段架构，不应在当前单节点上用进程复制模拟。
