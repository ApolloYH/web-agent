## 2. 产品定位与架构

这是一个类 ChatGPT 的个人超级助手。产品对外只有一个定位：帮助用户完成对话、写作、分析、文件处理和持续性的个人工作。

- **个人空间**：每个用户拥有独立配置、会话、技能、长期记忆和文件工作区。
- **多会话协作**：每个对话拥有独立上下文和运行状态，同一用户可以并行处理多个任务。
- **文件库与文档工作台**：统一管理对话文件和用户授权的本地目录，支持文档预览、编辑、问答和原位保存。

浏览器端 Mock、OpenAI 直连、Noumi 直连和 `/noumi-api` 代理已经删除，禁止按旧方案恢复。

## 3. 已完成能力

- 本地用户名密码登录、邀请码注册、显式管理员和 cookie session。
- SQLite 持久化用户、登录 session、对话、标题、消息和文件记录。
- 每个对话使用独立 Apollo Engine/session；同一用户可同时运行多个对话，侧栏显示运行状态。
- 每个用户拥有独立配置、固定 session、记忆、技能和文件工作区。
- `/clear` 清理当前上下文；会话自动压缩和 AI 标题总结已接入。
- Word、PDF、JPG/JPEG/PNG/WebP 对话内预览和文件库展示；预览区域限定高度并内部滚动。
- 消息复制、附件上传、对话重命名/移动/删除、侧栏运行旋转标识。
- 文件库支持 Word、Markdown、JSON 进入统一编辑工作台；PDF 和图片保持只读。
- Markdown/JSON 停止输入 350ms 后原位保存，JSON 保存前校验格式；保存串行排队，关闭文档前会强制等待最后一次写入。Word 关闭前会触发原生 DOCX 导出并等待回调。文件保存使用 ETag 防止覆盖外部新版本。
- Word 使用 `@agentbridges-ai/onlyoffice-browser` 纯浏览器编辑，不依赖 DocumentServer；本地 Office Host 使用独立 `127.0.0.1:5174` origin。
- 主站首屏 `load` 完成后会通过隐藏的 Office Host iframe 后台加载 9 个 Word 首开关键资源；Service Worker 将其缓存优先保存。构建会为主站 JS/CSS 和 Office 大文件生成与源码同步的 Brotli 文件，首屏主 JS 从 1.62 MB 降至 460 KB，Word 关键资源从 81.6 MB 降至 20.0 MB。
- Office Service Worker 只接管预热资源和打印 PDF，其余请求使用浏览器 HTTP 缓存；网络中断或编辑器销毁不会再产生无效 Response，缺失的高分屏格式图标请求与废弃 spellcheck 配置警告也已清理。
- Apollo 注册 `document_get_context`、`document_search_text`、`document_replace_text`、`document_append_text`、`document_set_content` 工具，通过现有 SSE/审批/响应链路实时操作当前浏览器编辑器。PDF 可提取文本和全文搜索，长文档先返回摘要片段再按关键词定位。
- 新增 Apollo Browser Bridge Chrome MV3 扩展，使用 `@page-agent/page-controller` 读取和操作用户明确选中的标签页；标签列表、打开/切换/关闭、点击、输入、选择和滚动全部复用现有 SSE 工具桥和审批。
- 新增可选独立 `browser-use` Worker 和 `browser_managed_task` 高风险工具，仅在配置 Worker URL 时注册；托管浏览器不共享用户 Chrome 的 Cookie。
- 桌面端左侧导航和文档右侧问答面板支持拖动调整宽度，也可聚焦分隔线后用方向键微调；宽度按用户保存在当前浏览器，窄屏继续使用抽屉布局。
- 对话内文档改为 GPT 风格大卡片，提供编辑、下载、全屏入口；文档写回后卡片预览同步刷新。
- 文件库可连接本地文件夹，目录句柄保存在 IndexedDB；Word、Markdown、JSON 的打开、手动编辑和写回都在浏览器中完成，原文件二进制不上传网站。若调用云端 Agent 理解或修改当前文档，文档正文会通过编辑器工具桥返回服务器/模型。
- CLI 与 Web 展示 thinking、工具调用和必要输出，完成后自动折叠。
- CLI 代码差异采用 `highlight.js`/`cli-highlight` 和 Monokai Extended 配色；Apollo Logo、输入提示和运行星形保持品牌蓝。
- 默认权限 `ask`；`readonly` 不可被 `--yolo` 绕过；Web 的全自动模式受 `WEB_ALLOW_UNRESTRICTED` 控制。无论是否启用全自动，Web 都永久拒绝系统命令、工作区越界、敏感文件读取、Skill 安装和 MCP 调用；普通用户的 MCP 配置会清空，Skill 目录固定在个人工作区。
- 正式 Node 生产服务、systemd、FRP、HTTPS 域名和持久化目录已建立。
- 服务端 Engine、轮次状态、session 路径和交互回执已改为按用户隔离；运行键包含用户 ID，跨账号可安全并行，同一交互只能由所属用户响应。
- 单节点默认限制为 8 个并发 Agent 任务、每账号 3 个，超限返回 `429 + Retry-After`；密码 scrypt 改为异步执行，避免并发登录阻塞事件循环。
- 生产服务新增 `/livez` 存活检查与 `/healthz` 就绪检查；就绪检查覆盖 SQLite、磁盘余量和 Office Host。响应带 `X-Request-ID`，Agent 流带 `X-Apollo-Run-ID`。
- 每次 Agent 运行写入 SQLite `agent_runs` 状态账本；服务重启会把遗留运行标为 `interrupted`，但不会自动重放可能产生外部副作用的任务。
- 默认保留 512MB 服务器空闲磁盘、每个用户工作区限制 2GB、上传文件保留 7 天；上传、产物导入和外部文件下载均有大小和落盘前校验。
- `scripts/backup-production.mjs` 使用 SQLite 在线 backup API 制作原子快照；`ops/web-agent-backup.timer` 每日触发。恢复脚本会保留切换前数据，完整步骤见 `docs/生产架构与恢复.md`。

## 6. Agent SDK 当前契约

Web 使用 `@apolloyh/apollo-agent@0.2.1`。关键宿主能力：

- `configMode: "isolated"`：只读显式配置，适合 Web 服务。
- `extraTools: ToolDefinition[]`：注册 Web 宿主专用工具。
- `memoryEnabled`：按用户配置启用或关闭记忆能力。
- `sessionId`：每个对话恢复自己的 session。
- `onEvent`：把 thinking、工具、审批、usage、文件和错误同步到 Web。
- `cancel()`：中断当前回合并回滚尚未完成的用户消息。
- `close()`：释放 MCP 和 Engine 资源。

完整说明见 `agent/docs/sdk.md`。

## 8. 已知限制与下一步

1. **当前仍是单节点架构**：多用户已可并发且状态隔离，但跨机器水平扩容仍需要把运行协调与 session 锁迁移到 Redis/队列，并把用户文件迁移到共享对象存储。
2. **Word Agent 目前是正文级修改**：支持读取、跨 run 精确替换、末尾追加并写回 DOCX；替换不再把整段内容压进首个 run，但表格结构、样式和版式级自动编辑仍需 OnlyOffice 插件桥。少数含复杂嵌入对象的 DOCX 会触发 X2T 兼容性错误，此时安全降级为只读预览且不改原文件。
3. **前端主包约 1.5 MB**：主要来自 DOCX 预览依赖，当前只是构建警告，不影响运行。
4. **PDF 与图片理解边界**：带文本层的 PDF 可读取和搜索；扫描件暂不做 OCR。图片当前只返回尺寸、类型和大小，尚无视觉内容识别。
5. **Web 沙箱边界**：应用层已永久禁用 Web 系统命令、越界和 MCP；生产仍必须保留 systemd 独立用户和沙箱，不能因应用层封锁而放宽操作系统权限。
6. **OnlyOffice 开源许可**：`@agentbridges-ai/onlyoffice-browser` 为 AGPL-3.0；主仓库已公开并加入 `AGPL-3.0-only` 许可，保留原有 ONLYOFFICE 品牌标识。
